# Обзор проекта rem_it_bot

Документ описывает архитектуру, функциональность и технологические решения проекта. Бот создан в период, когда не было своего домена, поэтому построен на long polling в одном PHP-процессе без веб-сервера и webhook.

---

## 1. Назначение проекта

**rem_it_bot (RIB)** — Telegram-бот «личный библиотекарь» для напоминаний.

- Пользователь присылает боту текст или картинку (цитаты, мысли, мотивацию).
- Бот сохраняет сообщения в БД и по расписанию отправляет их обратно по одному в день в заданный интервал времени.
- Идея: каждый день пользователь получает одну «поддержку от себя» из ранее сохранённого.

---

## 2. Архитектура и «топорная» технология

### Почему без домена и без webhook

- **Webhook** требует HTTPS и публичный URL. Нет своего домена — нет webhook.
- Выбран **long polling**: один скрипт в бесконечном цикле раз в N секунд запрашивает у Telegram API `getUpdates`, обрабатывает сообщения и проверяет расписание. Домена и веб-сервера не нужно.

### Как это устроено

| Компонент | Реализация |
|-----------|------------|
| Точка входа | Один PHP CLI-скрипт `main.php`, запускается в Docker-контейнере и крутится бесконечно |
| Получение сообщений | Каждую секунду (`PERIOD_MESSAGE_CHECKED`) вызывается `Processing::check()` → `$telegram->getUpdates()` |
| Расписание | Каждые 60 итераций — `Schedule::check()` (отправка по расписанию), каждые 600 — `Schedule::generate()` (построение расписания на день) |
| Хранение | MySQL (сообщения, расписание, очередь на день, «ожидающие» команды, история чата) |
| Файлы | Картинки сохраняются на диск в `app/file/{folder}/{filename}.jpg`, в БД — относительный путь |

Один процесс, один поток, последовательная обработка. Масштабирование «вширь» не заложено.

### Стек

- **PHP 8.2+** (CLI), **MySQL 8.2**
- **eleirbag89/telegrambotphp** — обёртка над Telegram Bot API (cURL)
- **MysqliDb** (thingengineer/mysqli-database-class) — работа с БД без ORM
- **Phinx** — миграции
- **vlucas/phpdotenv** — конфиг через `.env`
- **Monolog + Logtail** — логи в BetterStack
- **Sentry** — ошибки (опционально)
- **Docker Compose** — PHP CLI, MySQL, при dev — phpMyAdmin

## 3. Поток данных

```
main.php (цикл каждую секунду)
    │
    ├─► Processing::check()
    │       ├─ getUpdates() → для каждого update:
    │       │     ├─ команда /что-то → Action → command/Чтото.php → method
    │       │     ├─ редактирование сообщения → Message::edit()
    │       │     ├─ ответ на command_waiting → Action → нужная команда
    │       │     ├─ фото → Message::addImage()
    │       │     └─ иначе текст → Message::add()
    │       └─ addChatHistory() — пишем в chat_history
    │
    ├─► (каждые 20 итераций) heartbeat() → BetterStack
    │
    ├─► (каждые 60 итераций) Schedule::check()
    │       └─ getSendingDailyNow() → для каждой записи: getMessagePrepared() → sendMessage/sendPhoto
    │
    └─► (каждые 600 итераций) Schedule::generate()
            └─ getSchedulesNeedToday() → для каждого юзера: addSendingDailyNow() (N записей на день)
```

Роутинг команд: строка вида `/command/method` парсится в `Action` → подключается `command/Command.php`, вызывается метод `method()` (по умолчанию `index`). Команды получают только экземпляр `Telegram`; к БД обращаются через `new DB()` внутри себя.

## 4. Функциональность по модулям

### 4.1. Обработка входящих (Processing)

- Разбор текста: если начинается с `/` — считаем командой; специально обрабатывается `/_<id>` как «показать сообщение сейчас».
- Тип update: `edited_message` → редактирование сохранённого; `photo` → новое сообщение-картинка; иначе текст.
- Перед вызовом команды по тексту проверяется `command_waiting`: если пользователь «в режиме ввода» (например, настройки), следующий текст уходит в соответствующую команду (set_number, set_interval и т.д.).

### 4.2. Команды (command/)

| Команда | Описание |
|---------|----------|
| **Start** | `/start` — приветствие, создание дефолтного расписания (1 сообщение в день, 9–14, UTC+3). |
| **Message** | Добавление текста (`add`), картинки с подписью (`addImage`), редактирование в БД при edit в Telegram (`edit`), отмена по inline-кнопке (`cancel`). Дубликаты текста не сохраняются. |
| **Catalog** | `/catalog` — список сохранённых сообщений (короткие строки + `/_id`), разбивка по сообщениям по лимиту длины. |
| **Setting** | `/setting` — меню: интервал часов (например 9–20), количество в день, часовой пояс, «очистить все». Многошаговый ввод через `command_waiting`. |
| **Now** | `/now` или `/_<id>` — отправить сейчас следующее по очереди или конкретное сообщение по id. |
| **Change** | `/change` — изменить/удалить: подсказка про редактирование в Telegram, удаление по номеру, удаление последнего добавленного. |
| **Menu** | `/menu` — главное меню (inline-кнопки: каталог, настройки, FAQ, помощь). |
| **Help** | `/help` — контакт поддержки (@lyucean) + «версия» (дата изменения файла Help.php). |
| **Faq** | `/faq` — краткое описание идеи бота и советы по использованию. |
| **Error** | Неизвестная команда или ошибка — ответ пользователю; для одного захардкоженного admin chat_id показывается лог (legacy). |
| **Test** | `/test` — тестовые сценарии (если есть). |

### 4.3. Расписание (Schedule)

- **schedule** — настройки по пользователю: `hour_start`, `hour_end`, `quantity`, `time_zone_offset`.
- **schedule_daily** — план на текущий день: для каждого пользователя создаётся `quantity` записей со случайным временем в [hour_start, hour_end] с учётом offset (в UTC хранится).
- `generate()` раз в 600 итераций строит расписание на сегодня для тех, у кого его ещё нет и кто имеет сообщения.
- `check()` раз в 60 итераций выбирает записи `schedule_daily` с `date_time <= now()` и `status_sent = 0`, для каждой отправляет одно сообщение (текст или фото), помечает отправленным, старые отправленные подчищает.

### 4.4. Выбор сообщения для отправки

- По `chat_id` берётся одно сообщение: `display = 1`, минимальная `date_reminder`, затем `date_reminder` и `view` обновляются (очередь по кругу).

## 5. База данных (основные таблицы)

- **message** — сохранённые сообщения: `message_id`, `chat_id`, `text`, `image` (путь), `view`, `display`, `date_reminder`, `date_added`.
- **schedule** — настройки рассылки на пользователя: `chat_id`, `hour_start`, `hour_end`, `time_zone_offset`, `quantity`, `date_modified`.
- **schedule_daily** — слоты на сегодня: `chat_id`, `date_time`, `status_sent`.
- **command_waiting** — ожидание следующего ввода: `chat_id`, `command` (например `/setting/set_number`).
- **chat_history** — лог активности: `chat_id`, имена, username, текст, дата.

Миграции: Phinx в `app/db/migrations/` (стартовые таблицы, смена типа chat_id, utf8mb4).

## 6. Вспомогательные компоненты

- **config.php** — загрузка `.env`, обязательные переменные, пути `DIR_BASE`, `DIR_FILE`, `LOG_FILE`, Sentry, таймзона Europe/Moscow.
- **helper.php** — `get_var_query`, `shorten_line`, `shorten_text`, `shorten_link`, `is_url`, `ddf`, `heartbeat` (BetterStack; в коде используется `HEARTBEAT_TOKEN`, в конфиге может быть и `BETTERSTACK_TOKEN`).
- **logs.php** — синглтон логера Monolog с LogtailHandler (BETTERSTACK_TOKEN).

Картинки: скачивание с Telegram File API в `file/{folder}/{file_unique_id}.jpg`, при отправке — `curl_file_create` и `sendPhoto`.

---

## 7. Деплой и окружение

- **Локально/на сервере**: Docker Compose (php-cli, mysql, при необходимости composer, phpmyadmin). Контейнер php-cli стартует с `php /app/main.php`.
- **CI/CD**: GitHub Actions при push в `main`: сборка `.env` из secrets/vars, SCP файлов на сервер, SSH → `make update` (пересборка, composer, миграции), проверка контейнеров, уведомление в Telegram.
- Домен не используется: только long polling и при необходимости SSH/SCP на VPS.

---

## 8. Ограничения и техдолг

- Один процесс: при росте нагрузки нужна была бы очередь и/или webhook с несколькими воркерами.
- Роутинг через `include_once` и строковые имена классов; нет автозагрузки команд из контейнера.
- В каждом обработчике создаётся `new DB()` — общего Unit of Work нет.
- В Error.php зашит конкретный `TELEGRAM_ADMIN_CHAT_ID` для вывода лога — лучше вынести в конфиг.
- Таймзона зашита в Europe/Moscow в config и в генерации дат.
- Heartbeat в helper использует `HEARTBEAT_TOKEN`; в README/deploy упоминается и `BETTERSTACK_TOKEN` — стоит унифицировать имя переменной.

Документ даёт полную картину проекта и его «топорной» технологии без домена и webhook.
